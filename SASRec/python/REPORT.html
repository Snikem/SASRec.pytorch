Сравнение методов QAT для SASRec

INT8 fake-quantization, ml-1m, NDCG@10

Author

Anton

Published

April 26, 2026

1 Постановка задачи

Сравнить методы Quantization-Aware Training (QAT) для рекомендательной модели SASRec на датасете MovieLens-1M (ml-1m), метрика — NDCG@10 (вспомогательная HR@10).

1.1 Сравниваемые методы (все INT8)

Все методы реализованы вручную через torch.autograd.Function / nn.Module с STE для прохождения градиента.

2 Датасет и архитектура

Датасет: MovieLens-1M

  • users = 6040, items = 3416
  • средняя длина последовательности: 163.5
  • разбиение train/val/test — последовательное (последние 2 взаимодействия → val/test).

Архитектура (маленькая модель): hidden_units = 50, num_blocks = 2, num_heads = 1, maxlen = 200, ~212 K параметров.

Important

В SASRec ~80 % параметров — это item_emb. Если квантовать только FFN (как часто делают в учебных примерах), общий выигрыш по размеру будет мизерным, что мы и увидим в первом эксперименте.

Базовая FP32-модель: обучена с нуля 140 эпох (lr = 0.001, batch = 128), NDCG@10 ≈ 0.58 на тесте, плато достигнуто к эпохе 40.

3 Эксперимент 1. Квантизация только FFN

Что квантовалось:

  • forward_layers[i].conv1.weight, forward_layers[i].conv2.weight — веса FFN
  • активация между ними (ReLU → PACT/LSQ/APoT/DSQ; для AdaRound — uniform с running max)

Что НЕ квантовалось: эмбеддинги, attention, LayerNorm.

QAT-режим: 100 эпох дообучения от FP32 чекпоинта, lr = 1e-4, batch = 128, num_bits = 8. Каждый метод сохраняет свою модель в отдельную папку ml-1m_<method>_finetune/.

3.1 Результаты

Note

Про AdaRound: на диске .pth весит больше из-за обучаемого per-weight параметра \(\alpha\) (по элементу на каждый квантуемый вес). Эти параметры используются только во время обучения и в деплой не идут — поэтому deploy-размер считается без них.

3.2 Что увидели

  1. Все методы держат FP32 качество (даже немного лучше — до +0.7 % NDCG@10), потому что INT8 — это мягкая квантизация, и QAT её полностью компенсирует.
  2. Сжатие ×1.04 — разочаровывающее. Причина: FFN это всего ~5 % параметров; ×4 сжатие 5 % → ничтожный общий эффект.
  3. Скорости все близки к FP32; APoT и DSQ медленнее из-за tanh / lookup через torch.bucketize. Это fake-квантизация — реальные INT8-ядра CPU тут не используются, наоборот, к FP32-свёртке добавлены лишние операции.
Tip

Вывод первого этапа. Для видимого сжатия SASRec нужно квантовать не только FFN, а основные веса модели — в первую очередь item_emb.

4 Эксперимент 2. Квантизация всей модели

Что добавили в квантизацию (флаг --quant_full = true):

  • item_emb.weight, pos_emb.weight — квантизуем перед F.embedding
  • attention_layers[i].in_proj_weight и attention_layers[i].out_proj.weight — квантизуем перед вызовом F.multi_head_attention_forward

PACT остался без изменений — он по природе квантует только активации, для весов у него ничего нет. Это честное отражение метода: его сравниваем как activation-only baseline.

QAT-режим: 50 эпох (плато наступает к 40-й), всё остальное — то же. Результаты сохранены в ml-1m_full_<method>/.

4.1 Результаты

10 квантуемых тензоров = 2 × FFN.conv1 + 2 × FFN.conv2 + item_emb + pos_emb + 2 × attention.in_proj + 2 × attention.out_proj.

4.2 Что увидели

  1. Сжатие выросло до ×3.94 — это и есть видимый, ожидаемый эффект INT8 квантизации. До «честных» 4× мешают только LayerNorm/biases, которые остались FP32 (и так должны быть FP32).
  2. PACT даёт только ×1.04, потому что он не квантует веса в принципе. Для рекомендательных моделей с тяжёлыми эмбеддингами это ограничение PACT как метода.
  3. Лидеры по NDCG@10: LSQ (0.5896) → PACT (0.5885) → APoT (0.5883) → DSQ (0.5877) → AdaRound (0.5864). Разница между методами в пределах шума (≤ 0.005), но LSQ стабильно показывает лучший результат и при этом самый быстрый на CPU.
  4. На том же оборудовании квантуемые модели иногда даже быстрее FP32 (LSQ 126 ms vs 157 ms baseline) — это случайная вариация, fake-quant не должна давать ускорения.
Tip

Лучший метод по совокупности (качество + скорость + простота): LSQ. Его дальше и используем для финальной конвертации в реальный INT8.

5 Эксперимент 3. Реальная INT8-конвертация

Цель: превратить QAT-обученную модель в файл, где квантуемые веса физически хранятся как torch.int8 (1 байт на параметр), а не как FP32 с симуляцией квантизации.

5.1 Как делалось

Для каждого квантуемого тензора \(W\) известен его обученный scale \(s\) (от LSQ-квантизатора). Финальное представление:

\[ W_{\text{int}} = \text{clamp}\bigl(\text{round}(W / s),\ -128,\ 127\bigr) \in \mathbb{Z}_8 \] \[ W_{\text{deploy}} = W_{\text{int}}\cdot s \quad \text{(на инференсе)} \]

Сохраняемый файл model.int8.pth содержит:

  • int8 — словарь {имя тензора → int8 тензор}
  • scales — {имя → float scale}
  • fp32 — все остальные тензоры (LayerNorm, biases) как FP32
  • meta — гиперпараметры архитектуры

На инференсе строится обычный SASRec в FP32, квантованные веса распаковываются (int8 * scale) и кладутся в соответствующие позиции.

Скрипт: convert_to_int8.py. Бенчмарк: benchmark_int8.py.

5.2 Результаты на CPU

5.3 Что увидели

  1. Размер на диске упал в 3.8 раза (0.821 MB → 0.216 MB). Это и есть «настоящий INT8» — физически другие байты, не FP32 с эмуляцией. Небольшой недобор до ×4 за счёт служебных метаданных в .pth и biases / LayerNorm, которые остались FP32 (как и должны).
  2. NDCG у Real INT8 (0.5853) немного ниже чем у QAT fake-quant (0.5888), потому что при деплое мы «впекли» только веса: активационная квантизация (LSQ для активаций между conv1 и conv2) при деплое отброшена. То есть деплой-инференс использует FP32 ReLU, а QAT — INT8-активации. Эту активационную квантизацию можно тоже впечь — это даст +~0.003 NDCG, но усложнит код.
  3. Скорость ≈ FP32 базы (154 vs 163 ms). Это ожидаемо: deployed-инференс считает обычный FP32 matmul, просто веса предварительно распакованы из INT8. Чтобы получить реальное ускорение на CPU, нужны INT8-ядра (FBGEMM/QNNPACK), а под nn.MultiheadAttention PyTorch их не предоставляет из коробки — это отдельный объёмный кусок работы (замена MHA на quantizable-вариант, INT8 Conv1d, статическая квантизация по схеме prepare_qat_fx → convert_fx).

6 Сводное сравнение и выбор лучшего метода

Если оценивать качество + сжатие + простоту реализации, на маленьком SASRec на ml-1m лучшим методом QAT INT8 оказался LSQ:

  • лучший NDCG@10 (0.5896) среди всех методов в полном режиме квантизации;
  • даёт ×3.94 сжатие (предельно близко к теоретическому ×4);
  • самый быстрый CPU-инференс среди QAT-методов (126.6 ms);
  • единственная обучаемая величина — per-tensor \(s\), что упрощает анализ;
  • легко конвертируется в реальный INT8 (нужны только \(s\) и round).

Близкие к нему: APoT (0.5883), DSQ (0.5877), AdaRound (0.5864).

PACT ограничен в принципе — без квантизации весов сжатие модели в SASRec получить нельзя, потому что вес почти весь сидит в эмбеддингах.

8 Команды воспроизведения

cd /path/to/SASRec/python

# 1) FP32 baseline (если ещё не обучен)
python main.py --dataset=ml-1m --train_dir=default \
    --hidden_units=50 --num_blocks=2 --num_heads=1 \
    --num_epochs=140 --lr=0.001 --batch_size=128 --device=cuda

# 2) Все 5 методов в режиме full (квантизация всей модели)
DEVICE=cuda ./run_all.sh

# 3) Конвертация LSQ → реальный INT8
python convert_to_int8.py \
    --src ml-1m_full_lsq/SASRec.epoch=50.lr=0.0001.layer=2.head=1.hidden=50.maxlen=200.pth \
    --dst ml-1m_int8_lsq/model.int8.pth

# 4) Финальный бенчмарк FP32 vs QAT vs Real INT8
python benchmark_int8.py \
    --int8 ml-1m_int8_lsq/model.int8.pth \
    --fp32 ml-1m_default/SASRec.epoch=140.lr=0.001.layer=2.head=1.hidden=50.maxlen=200.pth \
    --qat  ml-1m_full_lsq/SASRec.epoch=50.lr=0.0001.layer=2.head=1.hidden=50.maxlen=200.pth \
    --dataset ml-1m --device cpu --threads 1 --num_runs 100

9 Краткий вывод

  • INT8 QAT на SASRec полностью сохраняет качество: все 5 методов держатся в пределах ±0.5 % NDCG@10 от FP32 baseline.
  • Видимое сжатие модели достигается только при квантизации эмбеддингов и attention — потому что в SASRec эмбеддинг item_emb весит 80 % параметров.
  • На реальном INT8 деплое получили сжатие ×3.80 при практически неизменном качестве (NDCG@10 0.5853 vs FP32 0.5806).
  • Лучший по совокупности метод для SASRec на ml-1m — LSQ.
  • Скорость на CPU не выросла — для ускорения требуются INT8-ядра, что для архитектуры с nn.MultiheadAttention требует отдельной значительной работы по замене модулей на quantizable-варианты.
